# Coding Standards — PTEPrep

This file is loaded by the **reviewer agent** during every code review.
It is also referenced by the **implementer agent** as the authoritative
standard to follow while writing code.

---

## Stack

- **Frontend:** React (Vite) in `apps/web` — TypeScript, functional components, hooks only
- **Backend:** Cloudflare Worker (Hono) in `apps/worker` — TypeScript, edge-native
- **Database:** Supabase (Postgres) with Row Level Security (RLS)
- **Storage:** Cloudflare R2 (private bucket) — media served via signed URLs only
- **Future:** React Native (Expo) in `apps/mobile`

---

## Style

- Use `camelCase` for variables and functions, `PascalCase` for types, interfaces, and React components
- Prefer named exports over default exports (exception: React page components)
- Use `const` by default; only use `let` when reassignment is necessary
- No `any` — use `unknown` and narrow, or define a proper type
- All async functions must have explicit return types
- Imports ordered: external packages → internal `@/` aliases → relative paths
- 2-space indentation, single quotes, trailing commas, semicolons (matches root `.prettierrc`)

---

## Testing

- Every public function and every Hono route handler must have at least one test
- Use red-green-refactor: write the failing test first, confirm it is red, then make it pass
- Test names must describe the expected behaviour: `"returns 401 when X-Admin-Key is missing"`
- Worker tests use Vitest + `@cloudflare/vitest-pool-workers`
- Web tests use Vitest + React Testing Library
- Run checks with `npm run typecheck --workspaces` and `npm run test --workspaces`
- Do not delete or modify existing passing tests unless the behaviour they test has intentionally changed

---

## Architecture

- **Never expose the Supabase anon key to the browser.** The Cloudflare Worker is the only caller of Supabase. The frontend calls the Worker, not Supabase directly.
- **Signed URLs only.** R2 media is served exclusively via signed URLs generated by the Worker (`/media/signed-url`). No public R2 bucket URLs ever appear in frontend code.
- **Signed URL TTL must be ≤ 3600 seconds.** Never set `expiresIn` higher than this.
- **Service role key stays in the Worker.** `SUPABASE_SERVICE_ROLE_KEY` must never appear in `apps/web`.
- Keep modules focused on a single responsibility (Ousterhout deep modules)
- Prefer composition over inheritance
- New Hono routes go in `apps/worker/src/routes/` — one file per resource
- Shared types between web and worker belong in `packages/` (future), not copy-pasted

## Security — OWASP Top 10 (2021)

The reviewer **must** check every diff against each item below and flag any violation as a blocking issue before approving.

### A01 — Broken Access Control

- Every Hono route that touches Supabase data must either (a) validate the Supabase JWT from the `Authorization` header and use the user-scoped client, or (b) validate `X-Admin-Key` for admin routes
- RLS policies are the last line of defence — they must never be the only line of defence
- Never use `service_role` key on routes accessible to end-users
- Check that no route leaks data from one tenant to another (tenant isolation via `tenant_id` column)

### A02 — Cryptographic Failures

- All secrets (`SUPABASE_SERVICE_ROLE_KEY`, `R2_SECRET_ACCESS_KEY`, `ADMIN_API_KEY`, `DEEPSEEK_API_KEY`) live in `.dev.vars` locally and Cloudflare Worker secrets in production — never in source code or committed `.env` files
- Signed URLs must use HTTPS endpoints only
- Never log API keys, tokens, or signed URLs

### A03 — Injection

- All Supabase queries must use the parameterised query builder (`supabase.from().select().eq()` etc.) — never string-interpolate user input into a query
- R2 object keys derived from user input must be sanitised: reject any key containing `..`, `/`, or `\` before passing to `getSignedUrl`
- Hono request bodies must be validated with Zod before use — never pass raw `c.req.json()` output directly to Supabase

### A04 — Insecure Design

- Auth tokens must be validated on every request — never trust a cached result without revalidating expiry
- The Worker must not implement business logic that should live in RLS (e.g., manually filtering by `tenant_id` instead of relying on RLS policies)

### A05 — Security Misconfiguration

- CORS in `apps/worker/src/index.ts` must whitelist only known origins (`localhost:5173` in dev, `pteprep.com.np` in prod) — never use `*`
- No debug endpoints or health routes that expose internal state (DB version, env vars, etc.) must be reachable without authentication
- The `Dockerfile` in `.sandcastle/` must not run as root — use the `agent` user

### A06 — Vulnerable and Outdated Components

- Do not add new `npm` dependencies without checking for known CVEs (`npm audit`)
- Prefer packages with active maintenance and fewer transitive dependencies

### A07 — Identification and Authentication Failures

- JWT validation must use Supabase's `auth.getUser(token)` — never decode the JWT manually without signature verification
- Admin routes must compare `X-Admin-Key` with `c.env.ADMIN_API_KEY` using a constant-time comparison where possible; at minimum reject immediately on mismatch without revealing which part failed
- Sessions must not be extended silently — let Supabase handle token refresh

### A08 — Software and Data Integrity Failures

- Blueprint JSON (the AI recipe column) must be validated against a schema before being stored or used to generate clones
- Never `eval()` or `new Function()` content received from the database or API

### A09 — Security Logging and Monitoring Failures

- Log all `401` and `403` responses with the route path and a timestamp (no user data in logs)
- Log all failed Supabase queries with the error code (no query parameters that might contain PII)

### A10 — Server-Side Request Forgery (SSRF)

- The Worker must never make outbound HTTP requests to URLs derived from user input
- Signed URL generation must only ever target the configured R2 bucket endpoint — never a URL constructed from request parameters

## Reviewer Checklist

Before approving any branch, the reviewer must confirm:

- [ ] All OWASP items above have been checked against the diff
- [ ] No secrets are hardcoded or logged
- [ ] All new routes have tests
- [ ] `npm run typecheck --workspaces` passes
- [ ] `npm run test --workspaces` passes
- [ ] No `any` types introduced
- [ ] Signed URL TTL ≤ 3600 seconds
- [ ] CORS origins are not widened to `*`
- [ ] No direct Supabase calls from `apps/web`
